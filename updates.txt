================================================================================
           EGGCELLENT EGGS: 6-PHASE PREPAID WALLET & DELIVERY LEDGER PLAN
================================================================================

PHASE 1: THE FINANCIAL FOUNDATION (WALLETS & TRANSACTIONS SCHEMA)
────────────────────────────────────────────────────────────────────────────────
Objective: Deploy the immutable financial engine and integrate it into the user profile.
1. Database Schema Execution:
   - Create `public.wallets` table to act as a fast balance cache.
   - Create `public.wallet_transactions` table to serve as the absolute financial source of truth.
   - Enforce check constraints on transaction types: ('recharge', 'delivery_deduction', 'refund', 'admin_adjustment', 'compensation').
2. Profile Integration:
   - Write a database trigger that automatically provisions an empty wallet (balance = 0.00) the moment a new user registers an account.
   - Build a "Wallet Balance" visual card inside the header navigation bar and the user account dashboard page.

PHASE 2: STOREFRONT SUBSCRIPTION CREATION & GROUPING MATRIX
────────────────────────────────────────────────────────────────────────────────
Objective: Align product families, set up fixed tier pricing, and build the template builder.
1. Product Hierarchy Upgrade:
   - Add an optional `parent_group_id` column to `public.products` to link family variations (e.g., 6-pack and 12-pack under a single family identifier).
2. Plan Management Refactor:
   - Update `public.subscription_plans` to hold fixed delivery parameters and explicit pricing modes: One-time (₹60), Subscription (₹50), VIP Subscription (₹45).
   - Incorporate `max_deliveries_per_week` and `is_vip_eligible` metrics directly into the plan template configuration.
3. Subscription Core Insertion:
   - Make the storefront "Subscribe" buttons save directly to `public.subscriptions` as an uncharged, paused, or pending template row defining only the user's delivery day choices, active status, and wallet processing mode.

PHASE 3: THE DELIVERY LEDGER ENGINE & THE CALENDAR GRID
────────────────────────────────────────────────────────────────────────────────
Objective: Build the date-granular delivery tracker and customer-facing calendar controls.
1. Ledger Database Deployment:
   - Create the `public.delivery_ledger` table to house individual daily delivery rows.
   - Map explicit statuses: ('scheduled', 'delivered', 'paused', 'skipped', 'failed', 'pending_payment', 'cancelled').
2. Customer Calendar View (`AccountSubscriptions.tsx`):
   - Build a monthly visual grid calendar where each date displays a color-coded dot mapping to the delivery ledger statuses for that user.
3. Future-Date Mutability Guardrails:
   - Allow users to tap a future date and click "Skip Delivery", changing that row's ledger state to `'skipped'`.
   - Enforce a hard application layer lock: Past dates are completely frozen, unclickable, and immutable forever.

PHASE 4: LOW-BALANCE GATEKEEPERS & RECHARGE FLOWS
────────────────────────────────────────────────────────────────────────────────
Objective: Connect payment gateways directly to wallet funding and handle insufficient balances.
1. Wallet Recharge Gateway Integration:
   - Build an in-app recharge portal where users select/input an amount and complete an online transaction (UPI / Card).
   - On payment success, log a positive row in `wallet_transactions` and increment the `wallets.balance` cache.
2. Checkout Gate Intercept:
   - Modify the progressive checkout handler. If a subscription is chosen, completely remove checkout cart processing. Replace it with a wallet verification step:
     * "Current Balance: ₹X. Minimum balance required to launch this contract: ₹Y."
     * Force a recharge redirect if the wallet balance sits below the configuration threshold.
3. Insufficient Funds Handling:
   - If a user's wallet is empty when a delivery is processed, flip that ledger row's state to `'pending_payment'`.
   - Read an admin configuration variable to determine if a single "Grace Delivery" is permitted before completely freezing subsequent shipments.

PHASE 5: THE DELIVERY PARTNER INTERFACE & FINANCIAL DEDUCTION GATES
────────────────────────────────────────────────────────────────────────────────
Objective: Reconstruct the partner portal around zero-cash validation and immediate ledger deductions.
1. Driver Manifest Transformation (`PartnerPortal.tsx` / `PartnerSubscriptions.tsx`):
   - Strip out all cash collection components, COD checks, and verification bills for subscriptions.
2. The Delivery Deduction Hook:
   - Replace the old completion buttons with a simple slide-to-confirm bar reading "Confirm Drop-off".
   - **Crucial Action Hook:** Sliding completion updates that specific `delivery_ledger` row to `'delivered'`. This database status transition fires a transaction trigger that automatically calculates the `effective_price`, appends a negative `'delivery_deduction'` row to `wallet_transactions`, and decrements the user's cached wallet balance. No deduction occurs if the delivery is skipped, paused, or marked failed.

PHASE 6: LOGISTICS LOGS & ADMIN MANUAL CLUSTERING BOARD
────────────────────────────────────────────────────────────────────────────────
Objective: Build the administrative control panels for manual route grouping and manual driver matching.
1. Manual Cluster Routing Panel (`AdminLogistics.tsx`):
   - Query tomorrow's `delivery_ledger` rows where status is `'scheduled'`.
   - Group rows into expandable visual buckets organized by Area, Pincode, and Apartment Name.
2. Bulk Partner Assignment:
   - Add a checkbox selection matrix next to the grouped delivery lists.
   - Admin selects 10, 20, or 50 delivery rows at once, hits a dropdown selection list, and clicks "Assign to Partner: [Driver Name]". This updates `delivery_partner_id` across those targeted rows simultaneously.
3. Overrides & Balances Adjuster:
   - Provide explicit admin override tools to manually adjust wallet balances (logging an `'admin_adjustment'` transaction type) and manually force-pause or force-resume a user's entire upcoming schedule during disputes.
